# 文档摄取系统 架构设计

概述
---
本系统接收用户上传的文件，使用可配置的并行分支调用 LLM 提取结构化数据，并将结果与任务元信息关联存储。设计目标：可扩展、可重试、可替换各模块实现、多分支并行处理。

高层架构（三层）
---
- 接入层（Ingress）
  - 用户认证（可插拔）
  - 文件上传 API：接收文件、生成 task_id（UUID）并返回给调用方
  - 将文件对象上传到文件存储（S3/OSS/本地），并把 metadata 写入任务表

- 处理层（Processing）
  - 读取分支配置（来自配置库）后对任务并行分叉，每个分支独立运行
  - 每个分支包含：指定 LLM、指定 Schema、验证与重试策略
  - LLM 调用：文件原始对象直接传给模型（不做预处理）
  - Schema 校验：验证 LLM 输出，失败时按照重试策略重试并将错误信息追加到 prompt
  - 重试仍失败：写入人工队列（人工处理）
  - 各分支互不干扰，某个分支失败不影响其它分支

- 存储层（Storage & Queues）
  - 配置库：保存分支配置、Schema、AI 参数（可使用数据库或配置存储）
  - 结果库：解析结果表（带 `task_id` 与 `branch` 字段），建议使用关系型数据库（MySQL/Postgres）或分析型（DuckDB）
  - 文件库：原始上传文件对象（S3、本地存储）
  - 人工队列：失败任务与错误信息，支持人工领取与回写接口


数据流（序列）
---
1. 用户上传文件到文件上传 API；系统返回 `task_id`。
2. 接入层将文件写入文件存储，并在任务表创建记录（状态：RECEIVED）。
3. 处理层读取分支配置（实时从配置库取），针对该 `task_id` 并行触发 N 个分支任务。
4. 每个分支：调用 LLM 进行解析 -> 校验输出与 Schema -> 成功则写入结果库；失败则按重试策略重试。
5. 若重试耗尽仍失败，分支写入人工队列并把分支状态标记为 FAILED。
6. 所有分支完成后，任务进入终态（部分分支可能失败）。


关键设计要点
---
- 无状态处理层：每次运行从存储层读取配置，处理层节点可水平扩展
- `task_id` 必须贯穿系统：任务表、结果表、人工队列均保留该字段
- 配置可热更新：分支数量、Schema、重试策略从配置库读取
- 重试逻辑：prompt 中追加上次校验错误信息，重试次数可配置；重试上限用以避免无限循环
- 分支隔离：分支间资源与错误隔离；单分支失败不影响整体吞吐
- 可替换 LLM：抽象 LLM 调用接口以支持多模型（OpenAI/GPT/Claude/内部多模态）
- Schema 验证：使用 JSON Schema 或自定义验证器，校验失败时生成可被人类理解的错误信息写入重试上下文


接口与示例
---
- 接入层 API（示例）
  - POST /upload
    - body: multipart/form-data 文件
    - response: { task_id: "uuid", status: "received" }

- 内部处理调度（示例）
  - POST /tasks/{task_id}/dispatch
    - 触发并行分支执行（从配置库读取）

- 结果写入（示例 schema）
  - results table columns:
    - id, task_id, branch_name, status, output_json, raw_llm_response, attempts, created_at

- 人工队列接口（示例）
  - GET /manual-queue?limit=20
  - POST /manual-queue/{item_id}/resolve


监控与可观测性
---
- 关键指标（指标收集）
  - 每秒/每分钟任务吞吐（TPS）
  - 分支成功率、失败率、平均重试次数
  - LLM 调用延迟与错误率
  - 人工队列积压长度

- 日志与链路追踪
  - 使用 `task_id` 作为 trace_id，确保整链路可追踪
  - 将 LLM 请求/响应、验证错误等敏感信息按策略脱敏或加密


运维与扩展建议
---
- 使用容器化（Kubernetes）部署处理层，利用水平伸缩应对突发流量
- 将 LLM 调用封装为异步任务，使用消息队列（RabbitMQ/Redis Streams）做缓冲与背压
- 对 Schema 仓库与分支配置进行版本管理，支持回滚
- 为人工队列提供简单的 Web UI，方便快速处理失败项


安全与合规
---
- 文件与结果存储应支持访问控制与加密（静态/传输层）
- 记录审计日志（谁触发、何时、结果）并定期清理临时敏感数据


结语
---
该文档基于 `AGENTS.md` 中的处理流程与约束整理，旨在为实现一个可扩展、可重试、可替换组件的文档摄取系统提供落地设计。
