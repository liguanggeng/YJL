## 系统一句话

用户上传文件 → AI 按 Schema 解析 → 结构化结果写入存储。
多用户，实时处理，出错可重试，重试仍失败转人工。

---

## 架构三层

### 接入层
- 用户认证
- 文件上传 API
- 生成 task_id（UUID），关联用户、文件、解析结果

### 处理层
- 按配置拆分为多个并行分支，每个分支独立运行  `[接口：分支数量和配置]`
- 每个分支：指定 AI + 指定 Schema，互不干扰
- 调用 LLM API 提取结构化数据（文件直接传，不做预处理）
- 对照 Schema 验证输出
- 验证失败 → 重试（prompt 追加错误信息）
- 重试仍失败 → 写入人工队列
- 某分支失败不影响其他分支

### 存储层
- 配置库：Schema、AI 参数、分支配置  `[接口]`
- 结果库：解析结果，带 branch 字段区分（MySQL 或其他）`[接口]`
- 文件库：原始文件对象存储  `[接口]`
- 人工队列：待人工处理的失败任务  `[接口]`

---

## 接口说明

以下模块设计为可替换，改动时只需换实现，主流程不动：

| 接口 | 当前假设 | 可替换为 |
|------|----------|----------|
| LLM API | Claude / GPT | 任意多模态模型 |
| 分支配置 | 待定 | 配置文件或数据库管理 |
| Schema 格式 | 待定 | JSON Schema 或自定义 |
| 结果存储 | MySQL | DuckDB、PostgreSQL 等 |
| 文件存储 | 待定 | S3、本地、OSS 等 |
| 人工队列 | 待定 | 数据库表、消息队列均可 |
| ID 生成 | UUID | 雪花算法（高并发时替换）|

---

## 数据流

```
用户上传文件
    ↓
接入层生成 task_id
    ↓
处理层读取分支配置（从配置库）
    ↓
并行分叉
    ↓               ↓               ↓
AI-A+Schema-A   AI-B+Schema-B   AI-C+Schema-C
    ↓               ↓               ↓
Schema 验证     Schema 验证     Schema 验证
    ↓               ↓               ↓
通过→结果库     通过→结果库     通过→结果库
失败→重试       失败→重试       失败→重试
仍失败→人工队列  仍失败→人工队列  仍失败→人工队列

所有结果共享 task_id，branch 字段区分
```

---

## 约束

- 处理层无状态，不持有任何配置，每次从存储层读取
- task_id 贯穿三层，所有表必须带此字段
- 重试时 prompt 必须包含上次的验证错误信息
- 新增文件类型无需改主流程，LLM 直接处理
